Input Dataset w/ a “Site” Column

    • If your data contains info on which site a fossil came from, and some fossils were found at the same site, we will randomize the ages by the common site, NOT by each fossil occurrence individually.
    • I.e., earlier we said extract.ages() and extract.ages.pbdb() (functions for parsing the raw data) would –> PyRate.py input file where each fossil's age was a random sample taken from within their individual, unique temporal age range (min_age to max_age)/
    • Now, we're saying that if we have data that specifies which site a fossil came from (a site column), and fossils share sites, we will instead assign each fossil occurrence an age that is a random sample of the SITE's unique temporal range. We'll average the min and max ages of all the fossils occurring at the same site, randomly sample within that, then assign EVERY fossil that occurred at that site the same age. So every fossil found at Site A will have the same randomly sampled age. Process:
  1. The average minimum age and maximum age of all fossils found in Site A will be calculated, and assigned as the temporal range for Site A, called 'mimima_1' and 'maxima_1' respectively
  2. The function runif() will randomly select an age from within each Site's range 'mimima_1' to 'maxima_1', round it to 6 decimal places, and store it as a new column (new entry in each array in *_PyRate.py) 'new_age'
  3. The resultant *_PyRate.py file will look the same as above except that each array (taxon/species) will only have a maximum of two (*caveat) values:
    1. Instead of multiple values of resampled ages for each occurrence of that taxon/species, only one value corresponding to that species’ site's randomly sampled age.
    2. *Caveat: If a species occurs in multiple sites, it will have a different random age for each site, so it will have one value for each site
    3. A 0 value at the end of the array, if it's an extant species (discussed previously)

Preservation Models:

  • Defining the preservation model for later use in the MCMC run
  • Epoch = a predefined time interval. Can be based on geological, environmental, etc. criteria. Set by the user in PyRate.
  • [IMAGE]
  • q rate = preservation rate
  • NHPP, HPP, TPP all deal with q rate temporal heterogeneity (or lack thereof, in the case of HPP) in different ways. However, they all assume all lineages have the same (possibly time-variable) q rate, which means they assume all lineages have equal chances of being preserved in the fossil record
  • Adding the Gamma model to these 3 models will introduce lineage-specific rates, taken from the gamma distribution
  • NHPP: Non-Homogenous Poisson Process of Preservation
    • Default model. Models q rate as a function of time Q(t) — a bell-shaped distribution where rates are lower at the two ends (origin and extinction of the lineage, which is what happens in the fossil record anyway). The rate's temporal heterogeneity is a continuous function, and is not defined by discrete epochs. All lineages have this same rate.
    • No additional flag is needed in your MCMC run
  • HPP: Homogeneous Poisson Process
    • Simplest model, assigns a constant q rate through all time for all lineage: a single parameter
    • -mHPP: add this to your MCMC run
  • TPP: Time-Variable Poisson Process
    • Rate heterogeneity through time, using discrete epochs rather than a continuous distribution like the NHPP
    • TPP explicitly introduces the idea of discrete epochs. Each epoch has its own individual, constant/unchanging preservation rate, allowing for abrupt changes in the preservation process at specific time points. TPP will estimate each preservation rate (and its individual prior probability distribution, more on that below) in each epoch independently. So, basically it's a piecewise compilation of HPP likelihoods across multiple time periods, each with their own preservation rate.
    • Create an epochs_q.txt file (or any other name), defining your epochs. No header, just one column with ages/time bins
    • -qShift path/to/epochs_q.txt -pP shape rate
    • PyRate will assign each time bin a gamma distribution, which can be thought of as the prior probability P(A) on that time bin’s q rate = how likely it was that that fossil was preserved, before taking into account the fossil occurrence data
    • It then randomly selects a q rate from each gamma distribution to assign as the q rate for that time bin
    • Using a probability distribution as the prior for each preservation rate allows PyRate to incorporate uncertainty and variability in the estimated q rates across different epochs
    • Gamma distributions have 2 parameters allowing the user to control its shape → control the prior probability → flexible framework to express prior beliefs while still letting the data inform posterior estimates
    • -pP (optional) controls those 2 parameters = shape and rate, both default to 1.5. This default assumes q rates are more likely to be concentrated around moderate values, with a lower robbaility of extremely high or low rates
      • Shape > 1 = unimodal (single peak) distribution. 1.5 = unimodal
      • Shape < 1 = exponentially decreasing distribution
      • Rate = scaling the gamma distribution = variability in q rates = changing the mean and variance of the preservation rates
        • Larger rate = distribution concentrated around smaller values
        • Smaller rate = more spread out distribution
      • -pP 0 0 → PyRate will estimate both parameters of the prior from the data
    • The model will use these priors as starting points/guides for the outputted preservation rates for each interval. These outputs = the posterior probability (P(A|B) = our model's updated belief about the preservation rates after we considered both
      • P(A) the prior probabilities (gamma distributions of the preservation rate in each epoch). How you set this obviously affects your posterior calculation, the ultimate output/goal of the model: P(A|B)
      • and P(B|A) the likelihood of observing the data (fossil occurrences)
  • Gamma Model: introducing Rate Heterogeneity Across Lineages to other models
    • -mG Added to any of the 3 Preservation models above! Not its own model
    • Applies a gamma distribution to each taxon’s preservation rate, so that each lineage’s q rate is chosen from that distribution (just like TPP’s process) → lineage-specific heterogeneity
    • The Gamma model for lineage heterogeneity only adds a single parameter, which voids over-parameterization
  • Gamma + TPP → Save Per-Lineage Relative Preservation Rates
    • -qShift path/to/epochs_q.txt -pP shape rate -mG -log_sp_q_rates
    • Save the estimated relative preservation rate (not s or e rate) for each lineage

  • We haven’t chosen a Birth-Death Model, but there are a ton of options talked about in Tutorials 2, 4, 5

    • -A 0 for parameter estimation, -A 1 for marginal likelihood/thermodynamic integration, -A 2 for BDMCMC, -A 3 for DPP (Dirichlet Process Prior)
      • All RTT graphs for -A 0 will be flat, because -A 0 performed 'parameter estimation'. You will see in your log files lambda_0 and mu_0, which are the constant rates. However, the mode -A 0 can also infer shifts when you combine it with -mL and -mM, respectively, and each set to a value > 1 (e.g. -mL means having two speciation rates and one shift; -mL 3 is three rates and two shifts).
      • Both BDMCMC (-A 2) and RJMCMC (-A 4, default) estimate the # and timing/placement of rate shifts (temporal rate heterogeneity). Therefore, they cannot be used with -fixShift. If you specify -A 2 or -A 4 with -fixShift, the model will default back to -A 0 every time
      • The difference is that they use different methods of proposing a rate shift. BDMCMC uses birth-death proposals, RJMCMC uses reversible jump proposal mechanisms
      • In practice, RJMCMC has been shown to work better than BDMCMC
      • See the -fixShift argument below
      • Note that any use of a BDNN model (a later tutorial) will lead to -A 0 every time as well
    • Preservation model flag:
      • No flag to default to NHPP
      • -mHPP for HPP
      • -qShift path/to/epochs_q.txt -pP shape rate for TPP, where -pP is optional
      • Gamma model flag -mG
    • Section 7. If you want to set fixed time bins for the BD model (BDC Skyline Model): do -fixShift path/to/bd_epochs_q.txt
      • bd_epochs_q.txt can be the same or diff from the epochs_q.txt in the TPP (I.e., can be the same file if you want to use the same time bins for Preservation and S+E (Birth-Death)
      • -fixShift cannot be used with -A = 2 or -A = 4. Calling -fixShift will lead to -A = 0, which is basic parameter (rate) estimation MCMC. This is because with -fixShift, you provide a file where you are specifically setting the # of timing of rate shifts. -A 4 (default), and -A 2 are MCMC algorithms that try to estimate the # and timing of rate shifts as their own parameters. Obviously incompatible with the user setting those values.

    • MCMC parameters:
      • -n 20000000 : do 20 million iterations or "steps" the MCMC algorithm will run (not 20 million states, depending on the acceptance/rejection rule). Default is 10 million! If the dataset is really large, Iterations may need to be increased in order to reach convergence
      • -s 5000: sample every 5,000. Default is 1,000! If the dataset is really large, you may need to sample the chain less frequently (increase -s) to reduce the size of the ouput files
      • n/s = the # of posterior samples the output log file will have.
      • So in this example, at 20 mill iterations, sampled every 5,000 –> the output log file will have 4,000 posterior samples
      • -p 100 = the program will show its progress every 100 iterations

Fixed Shifts at the Boundaries, While Searching for Rate Shifts Between Them

  • Sometimes fossil data is only available during a certain time window, which means the min and max boundaries of that time window SEEM like they have rate shifts, when they actually don't (this is the edge effect, where the data at the edges of the time bin reflect a sampling bias).
  • You can fix this by setting fixed times of rate shifts at the edges of that time bin (-edgeShift), where you know the dataset is cut off. You're basicaly setting temporal boundaries so you can truncate your analysis to just the rates within that time window.
  • python PyRate.py <data_set> -A 4 -edgeShift max min
    • max = a NUMBER representing the maximum age boundary
      • If you only want to set a maximum age boundary, just set min = 0
    • min = a NUMBER representing the minimum age boundary
      • If you only want to set a minimum age boundary, set max = to inf
    • -A 4: Then you run an RJMCMC on it to infer rate shifts within that time window

Running PyRate on Large Datasets

If you have 100’s-1000’s of lineages, MCMC will require a long time to reach convergence. PyRate can approximate a Gibbs sampler to more efficiently sample the ts’s and te’s

  • Gibbs only works for TPP or HPP preservation models
  • Gibbs vs. MCMC: two different types of MCMC algorithms
    • MCMC updates all params at once from their proposal distributions, accepting it according to an acceptance probability, so not every proposal is accepted
    • Gibbs approximates params’ full conditional posterior distributions (that param’s values holding all other variables constant), then samples the param’s value from that, skipping any sort of acceptance step
  • -se_gibbs -fU # # # → add this flag to your PyRate analysis run
    • -fU = update frequencies to set most of the updates (>90%) to focus on speciation, extinction, and preservation rates
    • q: .02, l/m: .18, cov: .08

Tutorial 5: (BD) Dispersal Extinction Sampling Models (DES)

  • DES does not include speciation process, but rather focuses on dispersal as the source of new lineages in an area. Birth = dispersal in DES
  • Each model will output the following predictions: dispersal rates (d12 from area A to B, and d21 from area B to A), extinction rates in each area (e1 and e2), preservation/sampling rates in each area (q1 and q2)
    • Normal MCMC output: posterior probs, model parameters
    • Additional DES model types will output additional predictions
  • This model assumes that present data is complete, because sampling probability at the present time is not modelled (and therefore only past time bins get incomplete sampling modelled).
    • In the code, this is implemented by setting the vector rho_at_present = 1 (present) if a species is present in that area in the most recent time bin, or 0 (not present), with no in-betweens (no uncertainty)
    • This makes the present distribution a “ground truth,” because all historical processes (dispersal, extinction, sampling) are inferred in a way that must lead to this known end state. More recent rates are most strongly influenced by recent time periods
    • This makes it important to use well-sampled recent occurrence data
  • For past data (fossils), the model accounts for incomplete sampling through heterogenous estimated preservation rates, q, just like normal PyRate does
  • Estimates biogeographic rates (dispersal rates and extinction rates) between 2 discrete areas (area-specific rates), whereas PyRate focuses on estimating s and e rates through time. So it’s biogeographic vs. temporal contexts
  • The geographic areas should generally be native species ranges (the areas where species naturally occur/are indigenous to rather than moved there) to avoid anthropogenic (human) influence

Preservation Rate Heterogeneity Across Taxa:

    • DES command above + -mG flag
      • -mG: just as in other analyses, -mG flag assigns different q (preservation/sampling) rates to each taxon from a Gamma distribution. It affects the preservation model! Sampling heterogeneity is computationally inexpensive in general, as it only adds a single free parameter to the model. Plus, it improves rate estimation. Therefore, -mG should always be included in DES models
      • -ncat 4: Mean sampling rate across all taxa = q, sampling heterogeneity = a discretized Gamma distribution with n categories. Default 4 Gamma categories is usually sufficient to account for heterogeneity across lineages. A higher number → more computation time.
        • What is “discretizing” a Gamma distribution? Chopping the distribution into 4 pieces, and taking the mean or median of each piece, and assigning each taxon into one of those 4 pieces. All the taxa in the lowest 25% of the Gamma distribution get assigned that 25%’s mean or median q rate, and so on. This is faster to compute than giving a unique q rate from the Gamma distribution to each taxon, while still allowing lineage-heterogeneity and approximating the continuous Gamma distribution

BDNN Tutorial

  • 2 feed-forward neural networks, one for speciation, one for extinction
  • Benefits of NN’s: does not need a priori assumptions about the shape, non linear and non-monotonic responses of rates to variation in predictors + accounting for interactions among predictors
  • Inputs: categorical (usually one-hot encoded, like geographic regions), continuous (environment, trait measurements), time is rescaled and added as a predictor by default
    • ^ Can take time data to get time-dependent rate changes without having to specify a prior discrete shift points
  • MCMC samples neural network params (weights), runs the FFNN with those weights, gets point estimates of rates, ts’s, te’s, which are then used to calculate species specific rates
    • Combines all those pt estimates from all those NN configurations to get distributions for those rates in the end
  • Process for one MCMC sample:
    • With each MCMC sample, only ONE neural net is updated: speciation or extinction
      • This is also why BDNN can’t be parallelized. One NN run through can only be done concurrently
    • For that neural net, either ts, te, q, OR one layer’s weights will be updated
    • If the weights are updated, then we re-run all the matrix operations from that layer onwards, through the newly configured neural net
      • The previous un-updated layers get cached, which speeds up the process relative to re-running the entire NN
    • We get new species specific rates
    • We calculate the new birth-death likelihood → new posterior distribution value
    • The posterior dist value that is sampled the most = the most likely configuration of the NN = the most likely ts, te, q, species-specific rates values
    • Also the point estimates of the predictions (ts, te, q, and rates) at each configuration of the NN (i.e., at each MCMC sample) lead to a probability distribution for the values of each ts, te, q, and rate
  • Time-series pedictors and time effects (s and e variance over time) are default binned in 1 myr time bins. So if you have a continuous temperature curve (-BDNNtimevar file), the model would use the average temp for each 1 myr breakdown of that curve → computationally efficient
  • If you use -fixShift and provide your own time bin file, you can specify when you want the s and e rates to vary
    • Then any -BDNNtimevar file with time-variable predictors would have their values averaged to fit w/in -fixShift’s bins
    • Benefit: maybe you want to delineate a larger bin to cover periods w/ fewer fossil occurrences → faster BDNN run
  • User can specify the # of layers and node
  • Activation function: ReLu or tanh in hidden layers, exponential function (to ensure positive rates) in output layer
  • Parameters: network weights, inferred with Bayesian MCMC → has uncertainty estimates
  • Predictions/Outputs: point estimates of speciation and extinction rates (events/million yrs) for that data + uncertainty estimates (?). Optional q (preservation) rates
  • The most important predictors can be ranked

Input: Examples (note that log Body mass and paleotemp are z-transformed)

  1. PyRate.py input file (created from occurrence data)
  2. Species-specific, unordered categorical and/or continuous, trait(s) data table following the -trait_file command. Rows = species (one per species), columns = traits
  3. Time-series/time-continuous predictors following the -BDNNtimevar command. Rows = time bins, columns = predictors

Data preparation:

  • Categorical with multiple traits → ohe
  • Categorical where species can be multiple at once → ohe but some will have 1’s for multiple columns
    • Analyze later with -BDNN_groups, re-combining into one group
  • Continuous traits and time-series predictors should be z-transformed (centering on 0) to improve model convergence
    • Continuous traits that DO NOT go negative should be log10 transformed first, then z-transformed
    • Their original mean and standard deviation should be stored in a Backscale.txt file so that when we later call the plotting function, the file can be used to display the results in their original scale
    • Time-series mean and std: first get both for each species through all time, then average those values for all species together → get mean and std → add it to Backscale.txt
  • Ordered categorical (ex: scale of 1-10 = 10 states where the order matters), should be centered on zero by subtracting the median value
    • Find the median, round up or down to the closest integer. Add that median to the Backscale.txt for use later when you call the -plotBDNN_effects flag, you can add the -plotBDNN_trans_features + this Backscale.txt file to make the plot more meaningful. Std = 1
    • Subtract that “median” from all values → centering roughly at 0
    • Then set std to 1 in Backscale.txt so that we can have integers for the body sizes
    • These centered values are not exactly the same as when using z-scaling (meaning their 95% interval is exceeding beyond +/- 1.96) but this should be fine for 7 categories. If the trait has more than 7 states, it should be better converted to a continuous trait.
  • 0 centering is important because our default activation function is tanh, which is symmetric around 0, [-1, 1] range. If your data is centered at 0, it makes full use of the range of tanh.
    • If your data’s mean was for example 2 instead of 0, most input would be in tanh’s saturation region (positive side), where the derivative is very small (flatter slope) and therefore gradients are very small

      • Example: if all of your species are from 200 million years ago and older, and are therefore all extinct: -translate -175.0 → shifts all the occurrences towards the present (removing 175.0 years from their ages). All time variable predictor files (-BDNNtimevar) will be adjusted automatically, and time re-scaling (-BDNNtimetrait) will now be more evenly distributed (instead of all the times being crunched around 0, they’ll be spread out more evenly)
      • This also speeds up the program, for the reason explained under -fixShift below
      • If you are working with completely extinct clades, always use -translate when:
        • You are NOT using -fixShift
        • You ARE including time as a trait, by using the default value for -BDNNtimetrait of -1 or any user chosen value >0

And this doesn't do any damage except slow things down/make the matrix bigger for no reason. So using -translate to shift the completely extinct species towards the future is not technically necessary as long as I am NOT including time as a trait (which means I'm using -BDNNtimetrait 0). But using -translate to shift the completely extinct species towards the future IS necessary if I DO include time as a trait (using -BDNNtimetrait -1, or >0, or just excluding it entirely, since the default value is -1).
    • -fixShift: Optional. One column, no header, time bins for the birth-death model, fixed times of rate shifts. If you do not use this, BDNN has default 1-myr time bins.
      • Including -fixShift time bins will make the program faster, but will not suggest to BDNN that there should be discrete rate shifts at those boundaries. The BDNN predictor “time” is something included by default (you would have to turn it off manually by using -BDNNtimetrait 0) can only capture gradual transitions in rates. Time as it is included as a trait using the default -BDNNtimetrait -1 will only affect the predictors (because all the predictors are interconnected/modulate each other in a neural network), but according to Torsten “will not be bent and twisted enough or have discrete shifts.” If you want to suggest sudden/discrete rate shifts at specific points in time, you would have to do some tricks:
        • Use RJMCMC instead of BDNN. RJMCMC excels in identifying shifts, but struggles with finding gradual transitions.
        • Provide a -BDNNtimevar file that contains time as rows, in whatever resolution (I think in Torsten’s example from the email, which you could probs find by searching “imptaient” or “truncated” or “refactor” he chose 0.1, which I think is 100k years??). And have one column per time bin you want, then a binary value of 0 or 1
          • ^ These are stage predictors, binary
        • However, if you have environmental trait files (following -BDNN_timevar) that have been binned into unequal intervals (like not all 5 myr or 1myr), it is better to provide -fixShift that matches those unequal -BDNN_timevar bins
      • Why is BDNN is painfully slow without -fixShift and -translate? Both are influencing how many matrix multiplications are performed by the neural network. By default we have one species x (traits + timevar + time) matrix per one million years. Each of these matrices is then multiplied by the weights in the first layer. The resulting dot product is multiplied with the weights of the next layer and so one.
      • The -fixShift allows to override the default 1 myr bins and reduces the number of matrices. The same when you do -translate because you get rid of 200 bins where you do not have fossil occurrences (this is all for the situation when we were working with completely extinct clades that went extinct 200 million years ago)
      • Also note that if later you want to bin/truncate PDP plots, the values by which you want to truncate the plots (values passed to the arguments -root_plot and -min_age_plot) need to be included in the fixShift file. If you want 1 Myr resolution but also want to bin/truncate the PDP plots later, you’ll have to create a -fixShift file that has 1 Myr resolution AND those years that you want to truncate the plots by. For example, if your analysis spans 300-200, and you want 1Myr bins, usually you could just leave out -fixShift since BDNN’s default time bins are 1Myr. But if you also later want to get PDP plots that don’t span the entire 300-200, but go 298.5-240, you need to make a fixShift file that looks like: 300, 299, 298.5, 298, 297…
    • -qShift: Optional. One column, no header, time bins specifying the TPP model. Can use any other Preservation model, TPP used here as an example
    • Should always use -mG: species-specific sampling heterogeneity from preservation model
    • -BDNNnodes 16 8: defines the number of layers and nodes in each layer. Default is 2 layers of 18 and 8 nodes respectively
      • If you have very few predictors, you should reduce the complexity of your network by reducing the # of nodes
      • -BDNNnodes 4 2 → if you have 3 predictors, for example
      • Also, if you reduce the # of nodes, you need to use the -BDNNupdate_f argument!
      • In the example of -BDNNnodes 4 2, you’d follow with -BDNNupdate_f 0.3
      • -BDNNupdate_f 0.3 is an increase from default 0.1 (which means that 10% of the weights in each layer are updated). If I did 0.1, since we have so few nodes, it’s possible that no weights would be updated.
      • With -BDNNnodes 4 2, you have:
        • Input layer to hidden layer: 1 * 4 = 4 weights
        • Hidden layer to output layer: 4 * 2 = 8 weights
        • Bias terms: 4 + 2 = 6 bias weights
      • The default -BDNNupdate_f updates 10% of weights per layer. This means:
        • Input layer: 10% of 4 weights = 0.4 weights (rounds to 0)
        • Hidden layer: 10% of 8 weights = 0.8 weights (rounds to 0)
        • Bias terms: 10% of 6 weights = 0.6 weights (rounds to 0)
      • This is problematic because with such small layers, rounding down means no weights get updated! You should increase the -BDNNupdate_f value to ensure at least one weight per layer gets updated in each MCMC iteration.
        • For example, setting -BDNNupdate_f 0.25 would update:
          • Input layer: 25% of 4 = 1 weight
          • Hidden layer: 25% of 8 = 2 weights
          • Bias terms: 25% of 6 = 1-2 weights
      • This would ensure the network can actually learn from the data.
    • -BDNNoutputfun: activation function of the output layer
      • 0) abs, 1) softPlus, 2) exp, 3) relu 4) sigmoid 5) sigmoid_rate (default=5)
      • Exponential is default, because it forces strictly positive output values. Since this function can grow very quickly, scaling and stability of your inputs and intermediate layers have to be well done
    • -BDNNactfun: activation function of the hidden layers
      • 0) tanh, 1) relu, 2) leaky_relu, 3) swish, 4) sigmoid (default=0)
      • Tanh is default. The benefit is that it allows for + and - interactions, bc it spans [-1, 1], and aoids vanishing gradients (relative to sigmoid). It was mainly chosen just because it has the best results though
    • -BDNNreg: affects regularization. BDNN automatically sets to 1.0 → assigns “no effect” (each predictor having no effect) the highest probability → the higher the positive values of a predictor’s effect, the larger the weight applied to it → a “shrinkage” factor to ouputted rates, pushing predictions towards a baseline of 1, which represents no effect → reducing overfitting w/ more conservative preditions
      • Conceptually = turning it higher means indicating the belief that extreme rate variations are less likely
      • -BDNNreg -1.0: turns reg off
      • -BDNNreg -1.0 1.0: turns reg off for speciation, on for extinction
      • -BDNNreg 1.0 2.0: default reg for speciation, higher reg for extinction, etc. for other combos

Predictor Importance: -BDNN_pred_importance

    • (1) Whether rate variation is correct:
      • Coefficient of variation for S and E rates: CV is a measure of dispersion in a dataset = std/mean. It is calculated for s and e rates, per species in each time-bin
      • ^This will not show up if you set -BDNN_nsim_expected_cv 0
      • BDNN allows rates to vary across species and time. Here we assess whether that variation is statistically significant, i.e., whether the BDNN variation is > the variation we’d expect to observe in a model where diversification rates were actually constant (the “null” hypothesis)
      • To do this, PyRate will create a constant rate model and feeds it multiple simulated datasets using the same root age and similar # of taxa (within 25% of the real dataset) as the real dataset
      • It will calculate the variation in rates for each simulated dataset’s run through that constant rate model
      • It determines the 95th percentile of these simulated rate variation = cv_expected
      • If the real data’s rate variation (cv_empirical) is > this 95% threshold, it suggests that the real data’s rate variation is > than expected by chance under a constant rate model → the variation in rates is statistically significant → BDNN model (or any model allowing for rate variation) is more appropriate than a constant rate model
    • (2) Ranking the importance of the predictors using 3 explainable AI metrics:
      • Marginal posterior probability of that predictor having an effect
      • Effect size of that predictor on the rates (SHAP values, can be + or -, larger in abs magnitude = stronger effect)
      • Effect of the predictor on model fit (feature permutation)
    • python PyRate.py -BDNN_pred_importance .../pyrate_mcmc_logs/*_mcmc.log -b 0.1 -BDNN_nsim_expected_cv 0
      • -BDNN_pred_importance_nperm: how many permutations, default 100 . Feature permutations will use the current time bins (set by the user in -fixShift and/or -qShift, or default 1 myr) and permute the values of each predictor within those bins, then see how the model’s likelihood changes. This is done several times per predictor, to see how important a predictor is to the model.
      • -BDNN_pred_importance_window_size: If your time bins are very small (like < 0.1 myr), the permutation (above) can take forever, since you’ll have to permute the values of each predictor within many bins multiple times. Set this to 0.5 to make the permutations happen in 500 kyr time bins, for example.
      • -resample: number of samples to keep from the log file to use for these calculations, if you have a ton of MCMC iterations, the calculation of predictor importance could take forever. So you’ll want to decrease this number
      • -BDNN_groups: optional, specify which predictors belong in the same class. Likely don’t need it unless you want pairwise comparisons of those groups. Like if you have a “biome” group with binary variables for 0 (not in that biome) vs. 1 (in that biome), and 5 biomes total, and you use -BDNN_groups, yo will end up with pred importance rows that compare the effect of presence in one biome to the effect of presence in another, rather than the effect of presence in one biome vs. absence in that same biome, which is more likely what you want.
      • -BDNN_pred_importance_interaction: rank 2-way interactions of predictors in addition to per-predictor. May slow things down depending on the number of predictors
        • -BDNN_mean_shap_per_group: use with -BDNN_groups to use the mean of the SHAPs instead of the sum of the SHAPs of all the members of each group to get overall importance of the group. Helpful when a trait has many states
      • -BDNN_nsim_expected_cv: # of simulations to use to get expected cv, coefficient of variation, default = 100, reduce it to make it faster

BDNN Tuning Suggestions:

    • Low number of traits = reduce model complexity.
    • -BDNNupdate_f 0.05 0.05 0.25, default 0.1
      • = what percentage of weights you want the network to update during each backpropogation. First set of weights = 3 predictors * 16 nodes = 48 weights. 0.05 of that = 2 weights updated instead of default 0.1 of that = 5 weights updated
      • Want the first layer of weights to be smallest since that has the most nodes
      • When too many weights are updated (especially of the first set), the BD_lik may decrease a lot and the proposed update will not be accepted. With many weights we could afford to update some of them because each will have a small leverage on the likelihood. Less updates mean the model is less explorative and may not find the highest likelihoods
    • Low ESS (high uncertainty): increase -BDNNupdate_se_f: same as above but for ts’s and te’s. Increases the share of proposed moves?
    • -se_gibbs: omits -mG and in addition samples origination and extinction conditional on the speciation, extinction, and sampling rate (it is slower though)

Non-BDNN Tuning Suggestions

Multimodality:

  • If ESS is super low for one run, check the marginal density plots of various TS’s and TE’s: it’s possible that they’re multimodal. This means PyRate is saying that there are 2 (or however many peaks there are) possible/likely times of speciation (that species might have originated at either point)
    • This can lead to bad ESS because the MCMC chain might not be taking a large enough jump from one peak to be able to get to the other. If it is taking too small a step, it might end up in the valley between the two peaks, which is a location/sample with low likelihood, implying a period of low speciation rate. Thus, the jump will be lower than the original starting point (one of the peaks), causing the move to be rejected
    • The default jump size/step size for each MCMC iteration is sampled from a uniform distribution of [-0.5,0.5] — this means each proposed jump is at MOST 0.5 Ma into the future or into the past from the current point
    • If you’re working with data where your two peaks/two potential times of speciation are 10 Ma apart, for example, it becomes really hard for any proposed MCMC jump to be accepted
    • When the chain finds one node/peak, it’ll get stuck exploring that region → consecutive samples are highly correlated → poor ESS
    • Trace plots can look like it has achieved stationatiry, but it might just be stuck at that one value without having properly explored both values
    • Marginal density plots may be multimodal, but the size of the modes is deceiving, bc it could just depend on where the chain got stuck the longest
  • Suggestion: increase step size with -tT argument. Generally aim for a step; size of 20-25% the distance between the two modes
  • General suggestions for MCMC (not specific to PyRate): use adaptive step sizes, implement parallel tempering to help chains jump between modes, use a mixture proposal distribution that occasionally makes larger jumps, run multiple chains with different starting points

  • BD1-1: Birth death model with constant speciation and extinction rates = -A 0
    • This is a legacy naming convention from the origins of PyRate when users had to manually set the number of rate shifts they wanted for s and e. When users manually chose 2 rate shifts for speciation and 3 for extinction, for example, the file would be:
      • BD2-1: BD model with 2 shift speciation rate and constant extinction rates = -A 0 -mL 2
      • BD3-2: 3 shifts in speciation rate, 2 shifts in extinction rate = -A 0 -mL3 -mM 2
    • Now they have -A 2 and -A 4 for inferring the # of rate shifts, so no need to set them manually
    • If you use -r -A will be forced to 0, so it will be MCMC, not RJMCMC (-A 4, default), and the output file will be named BD1-1
    • Same for -thread > 0?? According to the code that shouldn’t happen, but anytime I use -thread (and no -r), I still get output files with BD1-1

Note on extant species (use of the flag -N and how input data should be structured)

  • The sampling model can’t account for 100% complete sampling at the present, so you should NOT add in extant species occurrences that have no fossil record. I.e., you shouldn’t add in occurrence rows for presently living occurrences. Doing so would inflate rates and create edge biases. The program is only good for handling fossil data without present species diversity
    • The question we have as a result is, what if you have extant species? Obviously in the preparation of the data (the very first step using PyRate, the R poriton, where you use extract.ages) you can specify extant_species vector
    • But what if your occurrence dataset has NO fossil occurrences from the extant species? So you have a clade with fossil occurrences and extinct species, but that clade also has living species, but no occurrences of it? So if you specified that living, no occurrence species in the extant_species vector, nothing would happen. Can we add dummy occurrence data that has min_age = 0 and max_age = 0? No, as in the parent bullet point, you uhsould not
  • The -N flag is noted as an option in the original paper and in the program as a method of specifying the number of extant species in a clade, by going -N 8, for example. But Daniele Silvestro says that actually this doesn’t bring any improvements, so you should never use it.
  • What’s the solution for including present diversity in an analysis though? For PyRate, nothing really. Even changing the birth-death prior in order to make the shape reflect info about present diversity doesn’t really do anything to the estimates
    • Daniele says all we can do is “use the FBD model in a phylogenetic context e.g. using BEAST, which Gabriela and I discussed a while back as something interesting to try. These models are not perfect and might fail to pick up signals from ancient events like the K-Pg but might be better to detect accelerated diversification towards the recent”
